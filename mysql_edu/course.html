Getting Started

Introduction to MySQL

MySQL is the world's most widely used open-source relational database management system. It powers Facebook, Twitter, YouTube, and millions of web applications. Before you write your first query, you need to understand what MySQL is, how it works, and how to set it up.

What is a Relational Database?

A relational database stores data in structured tables — rows and columns. The “relational” part means tables can be linked through shared keys, allowing you to store data efficiently and query it with precision.

MySQL vs Other Databases: MySQL is one of many RDBMS options. PostgreSQL is more feature-rich. SQLite is embedded and file-based. Oracle is enterprise-grade. MySQL sits in the sweet spot: fast, free, widely supported, and battle-tested at massive scale.

Environment Setup

Installing XAMPP & Setting Up Your Workspace

XAMPP is the fastest way to get MySQL running locally. It bundles Apache, MySQL (MariaDB), PHP, and phpMyAdmin into one installer — no manual configuration needed.

1

Download XAMPP

Go to apachefriends.org and download the installer for your OS. Choose the version with PHP 8.x and ensure MySQL is included.

2

Run the Installer

Install to C:\xampp on Windows or /Applications/XAMPP on macOS. Ensure MySQL and phpMyAdmin are checked.

3

Start MySQL in the XAMPP Control Panel

Open the XAMPP Control Panel. Click Start next to MySQL. Status turns green — port 3306 is now listening.

verify port
# Windows: confirm MySQL is on port 3306
netstat -an | findstr 3306

# macOS / Linux
lsof -i :3306
4

Access phpMyAdmin

Open http://localhost/phpmyadmin. Default: username root, password empty. This is your visual database manager.

5

Connect via Terminal (Recommended)

terminal
# Windows
C:\xampp\mysql\bin\mysql.exe -u root -p

# macOS / Linux
/Applications/XAMPP/bin/mysql -u root -p

# After login, verify it works:
SHOW DATABASES;

Architecture

MySQL Server Internals

MySQL processes every query through a strict layered pipeline. Understanding each layer lets you debug slow queries, optimize schemas, and design production-grade systems.

The 6-Layer Architecture

Layer 1 — Client / Connectors (PHP, Python, Java, JDBC)
↓
Layer 2 — Connection & Thread Manager
↓
Layer 3 — SQL Parser & Preprocessor
↓
Layer 4 — Query Optimizer (Cost-Based)
↓
Layer 5 — Handler API
↓
Layer 6 — Storage Engine (InnoDB / MyISAM / Memory)

Layer Reference Table

LayerResponsibilityKey Component
ConnectorsLanguage drivers and APIslibmysqlclient, JDBC
Thread ManagerOne thread per connectionThread Pool Plugin
ParserLex and Yacc tokenizationParse Tree builder
OptimizerChoose best execution planCost-Based Optimizer (CBO)
Handler APIAbstract the storage layerha_innobase
Storage EngineRead and write physical dataInnoDB B-Tree files
Key Design Principle: MySQL separates query processing from data storage. The server layer does not care whether data is in InnoDB, MyISAM, or CSV — it communicates through the Handler API. This is why you can mix engines per table.
Architecture

The Query Life Cycle

Every SQL statement travels through this exact pipeline before returning a result. Knowing each step helps you understand why some queries are fast and others are slow.

1

Connection Established

MySQL authenticates the user against the mysql.user table. Assigns a thread. Loads privilege cache for the session.

status check
SHOW STATUS LIKE 'Threads_connected';
-- Variable_name       | Value
-- Threads_connected   | 3
2

Lexical Parsing

The SQL string is broken into tokens. Keywords, identifiers, and literals are classified into a Parse Tree.

token example
SELECT   -- keyword token
id       -- identifier token
FROM     -- keyword token
users    -- identifier token
WHERE    -- keyword token
age > 18 -- expression token
3

Optimization

The Cost-Based Optimizer evaluates all possible index paths. It picks the plan with the lowest estimated cost based on table statistics.

explain.sql
EXPLAIN SELECT * FROM users WHERE age = 25;
-- type: ref | key: idx_age | rows: 12 | Extra: Using index
4

Execution and Return

The Handler API calls the storage engine. The engine scans pages from the Buffer Pool or disk. The result set is returned to the client.

Normalization

1st Normal Form (1NF)

Rule: Every column must hold exactly ONE atomic value. No comma-separated lists, no arrays, no repeating groups.

BEFORE — Violates 1NF
IDNamePhones
1Sara555-1234, 555-5678
2John555-9999

Two phones in one cell — cannot index or query individually.

AFTER — 1NF Compliant
IDNamePhone
1Sara555-1234
1Sara555-5678
2John555-9999
1nf_schema.sql
CREATE TABLE user_phones (
  user_id  INT NOT NULL,
  phone    VARCHAR(20) NOT NULL,
  PRIMARY KEY (user_id, phone),
  FOREIGN KEY (user_id) REFERENCES users(id)
);

BCNF — Boyce-Codd Normal Form

Rule: For every functional dependency X → Y, X must be a superkey. This is a stricter version of 3NF for tables with overlapping candidate keys.

Joins Lab — All Join Types

Joins combine rows from two or more tables based on a related column.

Join TypeReturnsCommon Use
INNER JOINRows matching in BOTH tablesUsers with orders

all_joins.sql
-- INNER JOIN: only users who placed orders
SELECT u.name, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN: all users, NULL if no orders
SELECT u.name, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- Find users with NO orders
SELECT u.name
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;

-- SELF JOIN: find employee + manager name
SELECT e.name AS Employee, m.name AS Manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

Aggregation & GROUP BY

aggregation.sql
SELECT
  dept,
  COUNT(*) AS total_employees,
  AVG(salary) AS avg_salary,
  MAX(salary) AS top_salary,
  MIN(salary) AS lowest_salary,
  SUM(salary) AS payroll
FROM employees
WHERE active = 1
GROUP BY dept
HAVING AVG(salary) > 50000
ORDER BY payroll DESC;

Subqueries & CTEs

subqueries.sql
-- Scalar subquery
SELECT name, salary,
  salary - (SELECT AVG(salary) FROM employees) AS diff_from_avg
FROM employees;

-- IN subquery: users who placed orders
SELECT * FROM users
WHERE id IN (SELECT DISTINCT user_id FROM orders);

-- Correlated subquery: orders above user avg
SELECT * FROM orders o
WHERE total > (
  SELECT AVG(total) FROM orders
  WHERE user_id = o.user_id
);

-- CTE (Common Table Expression)
WITH high_earners AS (
  SELECT * FROM employees WHERE salary > 80000
)
SELECT dept, COUNT(*) FROM high_earners GROUP BY dept;

indexing.sql
-- Single column index
CREATE INDEX idx_email ON users(email);

-- Composite index (left-prefix rule applies)
CREATE INDEX idx_name_age ON users(last_name, first_name, age);

-- Unique index
CREATE UNIQUE INDEX idx_sku ON products(sku);

-- Full-text index
CREATE FULLTEXT INDEX idx_body ON posts(body);

-- Covering index: query only uses index, never touches table
CREATE INDEX idx_cov ON orders(user_id, status, total);
SELECT total FROM orders WHERE user_id=1 AND status='paid';
-- Extra: Using index ← never touched the table!

-- Show existing indexes

-- Remove unused index

EXPLAIN is the most important diagnostic tool in MySQL. It shows you exactly how the server will execute your query.

FROM users u
